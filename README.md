# Wishlist (Telegram WebApp)

Мини-приложение для ведения списков желаний в Telegram: веб-интерфейс (WebApp), бот для входа и уведомлений, серверная логика и администрирование доступа.

## Стек

- **Backend:** Python 3.10+, Django, **Django Ninja** (HTTP API, схемы на Pydantic v2), PostgreSQL  
- **Упаковка и качество:** Poetry; pytest, pytest-django; Flake8; mypy  
- **Интеграции:** HTTPX; при работе с Telegram Bot API — пакет **tg-api** (обёртка над HTTPX + Pydantic)  
- **Инфраструктура:** Docker / Docker Compose; CI — GitLab CI/CD  
- **Наблюдаемость:** **Rollbar** — сбор ошибок и стектрейсов (Django-бэкенд; при необходимости клиентский JS во фронте); ключи только через переменные окружения  
- **Асинхронность:** по мере внедрения — async-views/клиенты там, где уместно; учитывать ограничения Django ORM в async-контексте  
- **Frontend (отдельный этап):** React, **TypeScript**

Целевая архитектура — **несколько развёртываемых сервисов** (API, бот, фоновые задачи/уведомления и т.д.) с чёткими границами; на ранних этапах допустим один репозиторий и постепенное выделение контейнеров.

## План реализации (этапы и технологии)

Ниже — ориентировочный порядок работ и **какие темы освоить** в контексте WishList. Детали scope этапа можно уточнять в roadmap / issues.

### Этап 1 — Каркас API и конфигурация

- Django (проект, `urls`, settings), **Django Ninja**, Pydantic v2; при необходимости OpenAPI/Swagger для контракта с фронтом.
- **Переменные окружения:** `SECRET_KEY`, `DEBUG`, `ALLOWED_HOSTS`, строка БД; шаблон `.env.example`.
- **Poetry** — зависимости и запуск команд.

### Этап 2 — Данные и ORM

- **PostgreSQL:** типы, миграции Django, уникальные ограничения (инвайты, пользователи).
- **Django ORM:** модели, связи, транзакции (`atomic`) для сценариев вроде «активация инвайта + создание пользователя».
- **Django Admin** — выпуск и управление инвайтами на ранних итерациях.

**Изучение Django ORM:** курс в блоге **AISferaic** (серия «Курс Django ORM»). Рекомендуется проходить уроки **по порядку** от базовых блоков (модели, QuerySet, связи, транзакции, оптимизация). Урок по async-ORM: [Async в Django ORM — урок 9.3](https://aisferaic.ru/blog/tutorials/801/) — иметь в виду при **ASGI** и async-views; на **WSGI** достаточно синхронного ORM до появления реальной потребности в async.

### Этап 3 — Доступ и домен

- Модели **инвайтов** и активация; привязка к **Telegram user id**; политика одноразовых/многоразовых инвайтов.
- Минимальный **CRUD** вишлистов и желаний через Ninja API.
- **Аутентификация API:** проверка **Telegram WebApp `initData`** (HMAC) и/или сессия/JWT после активации — по выбранной схеме.

### Этап 4 — Качество и тестирование

- **pytest**, **pytest-django** — API и граничные случаи (просроченный инвайт, исчёрпан лимит, отзыв).
- **Flake8** или **Ruff**; **mypy** на бэкенде.
- Внешние HTTP (Bot API) — **моки** (`unittest.mock`, при необходимости **pytest-httpx**).

### Этап 5 — Telegram

- **Telegram Bot API**, **Mini Apps** / Web Apps, **HTTPX** и при необходимости **tg-api**.
- Deep links (`start` payload), кнопки WebApp, уведомления (в т.ч. под будущие напоминания).

### Этап 6 — Инфраструктура

- **Docker**, **Docker Compose** (API + PostgreSQL; позже — бот/worker).
- **CI** (GitLab CI/CD) — тесты и линтеры на merge.
- **Rollbar** — интеграция с Django (middleware/SDK), окружения `development` / `production`, отключение или заглушка в локальной разработке при необходимости.

### Этап 7 — Frontend

- **React**, **TypeScript** (например Vite); **React Router** при нескольких экранах.
- Запросы к API: **fetch** или **TanStack Query (React Query)**; типы — вручную, **Zod** или кодоген из OpenAPI.
- **Telegram Web App JS SDK** — тема, размеры, `initData` на клиенте (валидация остаётся на сервере).

### Этап 8 — Безопасность и продвинутые темы

- Хранение **секретов инвайтов** (хэш токена, как пароль); CORS / доверенные origin для WebApp.
- По мере роста нагрузки и фич (напоминания, массовые уведомления): **очереди** (например Celery + Redis) или аналоги — отдельное решение на этапе проектирования.

## Локальный запуск

Кратко (детали не дублировать здесь, если описаны ниже по файлам):

1. Установка зависимостей: `poetry install`  
2. Поднять PostgreSQL (например через Compose)  
3. Переменные окружения для БД и секретов — см. `.env.example` (без реальных секретов в репозитории)  
4. Миграции и сервер разработки — команды в разделе «Команды»

## Команды

| Действие        | Команда (пример)        |
|----------------|-------------------------|
| Тесты          | `poetry run pytest`     |
| Линтер стиля   | `poetry run flake8`     |
| Проверка типов | `poetry run mypy`       |

Точные пакеты и скрипты — как зафиксировано в `pyproject.toml`.

## Соглашения по разработке

- **Frontend:** React + TypeScript (отдельное приложение); типы контрактов с API — по OpenAPI/Swagger бэкенда или сгенерированные клиенты, по мере готовности.  
- **API:** новые эндпоинты — через Django Ninja; вход/выход описывать схемами Pydantic v2.  
- **Данные:** миграции Django обязательны при изменении моделей.  
- **Тесты:** критичная логика и API покрываются pytest/pytest-django; внешние HTTP — моки (`unittest.mock`, при необходимости `pytest-httpx`).  
- **Типы:** код по возможности типизирован; mypy в CI или перед merge.  
- **Секреты:** не коммитить токены и пароли; только шаблоны в `.env.example` (в т.ч. placeholder для **Rollbar** server access token).  
- **Микросервисы:** перед выносом функционала в отдельный сервис — зафиксировать контракт (HTTP схема/очередь) и зависимости в этом README или в `docs/`.

## Продуктовый контекст (кратко)

- **Доступ:** основной механизм — **инвайты** (код или одноразовая/многоразовая ссылка по политике продукта); выпуск, лимиты и отзыв — через **админку**; бот/WebApp используются для входа уже после (или в момент) активации инвайта.  
- Дальнейший функционал: вишлисты, ссылки и ручные поля к желаниям, календарь/события, бронирования (в т.ч. анонимные), напоминания по правилам пользователя, превью по ссылкам (best-effort), фильтры. Реализация идёт итерациями; актуальный scope этапа — в roadmap / issues.
